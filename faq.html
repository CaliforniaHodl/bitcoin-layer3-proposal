Hard Questions, Honest Answers

The answers come from the code, not from ideology.

How far did Core 30 drift from Satoshi's blueprint? ▾

Massively in one specific dimension: data capacity. 100,000x increase from ~40 bytes to ~4,000,000 bytes per block, at a 75% discount, with no content validation. The blueprint said "scalpel." Core 30 says "fire hose."

Three compound changes: SegWit's witness discount (2017), Taproot's script size exemption (2021), and unknown leaf version acceptance (2021). Each reasonable alone. Together they built the inscription highway.

Knots sees the drift and tries to correct it, but only at the policy layer (mempool), not consensus (block validity). Every rejection uses TX_NOT_STANDARD, never TX_CONSENSUS. Knots doesn't fork. It just doesn't help relay what it considers spam.

Are Core devs trying to destroy Bitcoin from the inside? ▾

No. That's civil war drama. The code evidence doesn't support it.

Every decision had a legitimate engineering reason. SegWit's discount was needed for malleability fix adoption. Taproot's script size exemption was for future extensibility. Unknown leaf version acceptance is standard protocol engineering for forward compatibility.

The fair criticism isn't "they're destroying Bitcoin." It's:

"They optimized for protocol evolution and didn't constrain for misuse."

They built a highway on-ramp for future features and didn't put a weight limit on the vehicles. Someone showed up with a convoy of dump trucks full of JPEGs. That's a design gap, not an attack.

"Never attribute to malice what is adequately explained by compounding side effects across three soft forks over four years."
Can inscriptions be stopped? ▾

Wrong question. The right question is: can inscriptions be made irrelevant?

Bitcoin L3 anchors documents to L1 using a 36-byte OP_RETURN — not by stuffing 4 MB JPEGs into witness data. Same proof of existence. Same immutability. 10,000x less chain bloat. With batch anchoring via Merkle trees, you can anchor 1,000 documents in a single transaction for ~440 sats total.

The political fight over MAX_TAPSCRIPT_SIZE or witness discounts doesn't need to be won if L3 makes inscriptions look expensive and stupid by comparison. Make the alternative so obviously better that inscriptions die from irrelevance, not restriction.

Policy-level filtering (Knots-style) still works for individual nodes. But the real fix isn't banning data from L1 — it's building a proper place for it on L3.

Why can't we just ban data storage on Bitcoin? ▾

Because the security budget might depend on it. Inscriptions generate fees. Fees pay miners. Miners provide security. As the subsidy halves toward zero, every revenue source matters.

The answer isn't banning data — it's moving it off-chain. L3 anchoring still generates fee revenue (every anchor is a transaction), but each anchor is 36 bytes instead of 4 MB. Miners still get paid. Full nodes don't store everyone's JPEGs. Data users get better tooling — encryption, batch anchoring, P2P distribution, content policies.

Fair pricing helps too: reduce the witness discount from 4:1 to 2:1, cap script sizes at 100KB. But even without L1 changes, L3 makes the economics obvious. Why pay for 4 MB of witness space when 36 bytes does the same job?

What about the "inscriptions are free speech" argument? ▾

Bitcoin is a protocol, not a constitution. Protocols have technical constraints. The 40-byte OP_RETURN limit was always a constraint on what you could store. Nobody called that "censorship" in 2009.

Setting MAX_TAPSCRIPT_SIZE = 100KB doesn't censor anything. It caps how much data fits per script. You can still anchor data. You can still timestamp. You just can't store 4MB JPEGs at a 75% discount. The free speech argument confuses access to a subsidized resource with a fundamental right.

Is the Impossible Pentagon solvable? ▾

Not fully. That's why it's called "impossible." You can't maximize all five properties simultaneously without the subsidy acting as universal lubricant.

But you can reach equilibrium — a configuration where everyone gets enough, even if nobody gets everything. The six compromises in this research map out one viable equilibrium point. There may be others. The important thing is that someone designs one, because the current plan is no plan at all.

What does PFM stand for? ▾

Pure Fucking Magic.

PFM is a section-based document container with cryptographic integrity, lazy I/O via byte-offset indexes, and content-addressed deduplication. The checksum of a PFM document becomes its filename in the L3 store and the value anchored to Bitcoin. Any modification to any section changes the checksum, breaking the anchor chain.

Is Bitcoin L3 real code or just a whitepaper? ▾

Real code. 18 CLI commands. 287 test functions. 276 passing. 4 rounds of adversarial security testing. 43 security fixes. Zero critical or high vulnerabilities remaining.

Install it with pip install -e . and run l3 store, l3 anchor, l3 verify-anchor today. It works on regtest, testnet, and mainnet.

What is VeilCloud? ▾

VeilCloud is the native privacy layer built into Bitcoin L3. It provides:

  • Client-side encryption — AES-256-GCM, so the network never sees plaintext
  • Threshold key management — Shamir’s Secret Sharing over GF(256). Split a key into N shares, any T recover it, fewer reveal nothing
  • Batch anchoring — Merkle trees let you anchor thousands of documents in a single 36-byte OP_RETURN. Each document is independently provable via inclusion proof
  • Audit trails — append-only hash-chained logs with Merkle proofs
  • Access control — HMAC-signed credentials with permissions, expiry, and revocation

All pure Python. The only optional dependency is cryptography for the encryption module. Everything else is stdlib.

What does it cost to anchor a document? ▾

An OP_RETURN output adds ~44 vbytes. At 10 sat/vbyte, that’s roughly 440 sats (~$0.40 at $100K/BTC) for a single document.

With batch anchoring via Merkle trees, you can anchor 1,000 documents in one transaction for the same 440 sats. Each document is independently provable via its Merkle inclusion proof. That’s less than 0.5 sats per document.

What is Bitcoin Finney? ▾

A minimal Bitcoin node for L3 anchoring, named after Hal Finney. It’s a stripped-down fork of Bitcoin Knots that retains only bitcoind and bitcoin-cli with SQLite wallet support.

No GUI, no ZMQ, no Tor, no UPnP, no BDB wallets, no utility binaries, no test suites. Two binaries. Full node. Full RPC. That’s it.
