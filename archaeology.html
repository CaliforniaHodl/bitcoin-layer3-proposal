The Baseline: What Satoshi Built

Bitcoin v0.1.0 was a carefully balanced system designed as peer-to-peer electronic cash with deliberate constraints against data bloat. Every limitation was intentional.

OP_RETURN: The 40-Byte Scalpel

Satoshi recognized the need for small data anchoring but constrained it to prevent bloat. The 40-byte limit was chosen to allow timestamping and commitments while preventing the chain from becoming a data store.

core0.1/bitcoin/src/script/script.h:65 C++ 
OP_RETURN = 0x6a
core0.1/bitcoin/src/script/script.h:590-593 C++ 
bool IsUnspendable() const {
    return (size() > 0 && *begin() == OP_RETURN);
}

Marks outputs as permanently unspendable. The UTXO set immediately prunes these. Limited to 40 bytes by default. Only one per transaction. Configurable via -datacarriersize.

The Fee System

core0.1/bitcoin/src/main.cpp C++ 
CAmount GetBlockValue(int nHeight, const CAmount& nFees) {
    CAmount nSubsidy = 50 * COIN;
    int halvings = nHeight / SubsidyHalvingInterval();  // 210,000 blocks
    if (halvings >= 64) return nFees;
    nSubsidy >>= halvings;
    return nSubsidy + nFees;
}
The design philosophy: everything is constrained, everything is validated, data storage is a footnote.

This is the baseline. Everything that follows is divergence.

The Divergence: Three Decisions That Built a Highway

Three engineering decisions, each reasonable in isolation, compounded into a structural change nobody voted for. The inscription pathway requires all three—remove any one and inscriptions become significantly harder.

2009
Satoshi v0.1
1MB blocks, 40-byte OP_RETURN, no witness, fee-by-byte. Peer-to-peer electronic cash.
2014
OP_RETURN Expanded
40 → 80 bytes. Minor change. Still constrained.
2017
SegWit (BIP 141) — First Major Divergence
Witness data structure with 4:1 weight discount. Block capacity 1MB → ~4MB. Fee calculation changes to weight-based. Intent: Fix malleability, enable Lightning. Effect: Created cheap data lane with no content validation.
2021
Taproot (BIP 341) — Second Major Divergence
Script size limit removed for tapscript. Unknown leaf versions accepted without validation. Intent: Better privacy, future extensibility. Effect: Unlimited data at 75% discount, no execution required.
2023
Ordinals / BRC-20
Inscription boom. ~4MB of JPEGs per block. The "Merchant's Discovery."
2024
Knots Filtering
Policy-level filtering. Mempool only. Blocks still accepted. The "Operator's Response."
2025
The Civil War
Taproot fully mature. Inscription pathway well-established. The debate rages.

The Three Exact Code Points

Point 1: The Witness Discount

core22/bitcoin-22.0/src/consensus/consensus.h:21 C++ 
static const int WITNESS_SCALE_FACTOR = 4;

One line. 4:1 ratio. Witness data becomes 75% cheaper than non-witness. Designed to incentivize SegWit adoption for signatures. Created an economic incentive to put any data in witness format.

Point 2: The Script Size Exemption

core30/bitcoin/src/script/interpreter.cpp:428 C++ 
if ((sigversion == SigVersion::BASE || sigversion == SigVersion::WITNESS_V0)
    && script.size() > MAX_SCRIPT_SIZE) {
    return set_error(serror, SCRIPT_ERR_SCRIPT_SIZE);
}
// Tapscript (SigVersion::TAPSCRIPT): NOT CHECKED

Two lines missing. Tapscript was exempted from the 10KB script size limit. MAX_SCRIPT_SIZE = 10,000 applies to legacy and witness v0. Tapscript has no script size limit at consensus level.

Point 3: The Unknown Leaf Version Pass-Through

core30/bitcoin/src/script/interpreter.cpp:1982-1992 C++ 
if ((control[0] & TAPROOT_LEAF_MASK) == TAPROOT_LEAF_TAPSCRIPT) {
    // Execute as Tapscript (only leaf version 0xc0 gets executed)
    return ExecuteWitnessScript(...);

// UNKNOWN LEAF VERSIONS: accepted without any validation

    return set_error(...);  // Policy only - not consensus
}
return set_success(serror);  // <-- THE VULNERABILITY

Three lines. Unknown leaf versions accepted without any content validation. Any data can be committed as a "script" with an unknown leaf version. The data is never parsed, never executed, never validated. It sits in the witness forever.

The Filtering: What Knots Does and Doesn't Do

Bitcoin Knots implements policy-level filtering of inscriptions and data-carrying transactions. It will not relay or mine them, but it will accept blocks containing them. This is NOT a consensus change—Knots and Core stay on the same chain.

The Critical Answer: Policy, Not Consensus

All Knots filtering is mempool-only. Evidence from the code:

  • validation.cpp:850 — Rejections use TX_NOT_STANDARD (policy)
  • validation.cpp:1030 — Data rejections use TX_INPUTS_NOT_STANDARD (policy)
  • Knots accepts the block even if it contains inscriptions

std::pair<size_t, size_t> CScript::DatacarrierBytes(const size_t remaining_outputs) const
{
    if (size_t olga_bytes = IsOLGA(remaining_outputs); olga_bytes) {
        return {0, olga_bytes};
    }

    size_t counted{0};
    opcodetype opcode, last_opcode{OP_INVALIDOPCODE};
    unsigned int inside_noop{0}, inside_conditional{0};
    // ...
    // Match OP_FALSE OP_IF ... OP_ENDIF (dead code = data)
    if (opcode == OP_IF && last_opcode == OP_FALSE) {
        inside_noop = 1;
        data_began = opcode_it;
    // Match <data> OP_DROP
    } else if (opcode == OP_DROP && last_opcode <= OP_PUSHDATA4) {
        counted += it - data_began;
    }
    return {0, counted};
}

CalculateExtraTxWeight(): The Penalty System

knots/bitcoin/src/policy/policy.cpp:584-608 C++ 
int32_t CalculateExtraTxWeight(const CTransaction& tx,
    const CCoinsViewCache& view, const unsigned int weight_per_data_byte)
{
    int32_t mod_weight{0};
    if (weight_per_data_byte > 1) {
        for (const CTxIn& txin : tx.vin) {
            const CTxOut &utxo = view.AccessCoin(txin.prevout).out;
            auto[script, consensus_weight_per_byte] =
                GetScriptForTransactionInput(utxo.scriptPubKey, txin);
            if (weight_per_data_byte > consensus_weight_per_byte) {
                const auto dcb = script.DatacarrierBytes(0);
                mod_weight += (dcb.first + dcb.second) *
                    (weight_per_data_byte - consensus_weight_per_byte);
            }
        }
    }
    return mod_weight;
}

Instead of rejecting, Knots can penalize data-heavy transactions by inflating their virtual weight. Configured via -datacarriercost (default: 1.0). A value of 10 makes data bytes cost 10x normal weight.
