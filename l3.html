Architecture: Anchors on L1, Data on L3

Bitcoin L3 separates concerns. Documents live off-chain in a content-addressed store with peer-to-peer distribution. Bitcoin L1 holds only a 36-byte anchor — a 4-byte protocol prefix and a 32-byte SHA-256 checksum embedded in an OP_RETURN output. The anchor is permanent. The data is portable. L1 proves existence. L3 provides availability.

Bitcoin L3 ArchitectureDesign
    L1 (Bitcoin)              Bridge              L3 (Document Layer)

   |                 |    |             |    |                        |
   |  Block N        |    |   36 bytes  |    |  ~/.pfm/l3/store/     |

   |  | OP_RETURN |  |<---|  PFM3 +     |<---|  | <checksum>.pfm   | |

   |                 |    +-------------+    |  +------------------+ |
   |  Immutable.     |                       |                        |

   |  Proves time.   |                       |  Content-addressed     |

L1: The Trust Anchor

Bitcoin L1 serves one purpose: timestamped proof of existence. For each document, a single OP_RETURN transaction is broadcast containing:

On-Chain Anchor FormatProtocol
Script:  OP_RETURN OP_PUSH(36) <PFM3> <SHA-256>
Hex:     6a24 50464d33 <64 hex chars>
Size:    38 bytes scriptPubKey (2 opcode + 36 data)
Cost:    ~440 sats at 10 sat/vbyte (~$0.40 at $100K/BTC)

Once confirmed, this anchor provides:

  • Proof of existence — the document existed at block height N
  • Tamper evidence — any modification changes the checksum
  • Non-repudiation — the anchor cannot be removed or altered
  • Timestamping — block timestamp provides approximate dating

L3: The Document Layer

All document data lives on L3:

  • Local store — content-addressed filesystem at ~/.pfm/l3/store/
  • Index — index.json maps checksums to metadata (txid, network, timestamps)
  • P2P network — nodes discover each other via Nostr relays and exchange documents
  • PFM format — a purpose-built container format with cryptographic integrity
L3 is where the data lives. L1 is where the proof lives. The bridge between them is a 36-byte OP_RETURN.

The PFM Container Format

PFM (Pure Fucking Magic) is a section-based document container designed for content-addressed storage.

PFM Document StructureFormat
#!PFM/1.0
#@meta
id: a1b2c3d4-...
agent: my-application
created: 2026-02-24T12:00:00Z
checksum: sha256:<hex>
#@index
content 247 1024
attachment 1271 8192
#@content
The actual document content goes here.
#@attachment
<binary or text data>
#!END
PropertyMechanism
IdentitySHA-256 checksum of all section content
IntegrityChecksum embedded in metadata, verified on read
DeduplicationContent-addressed — identical content = identical checksum
Lazy I/OByte-offset index enables O(1) section access without reading entire file
PortabilityText-based format with escape sequences for binary safety
Size limits100 MB max file, 10,000 max sections

P2P Document Network

Documents are useless if they can't be found. Bitcoin L3 includes a peer-to-peer network for document discovery and exchange.

Peer Discovery via Nostr

Nodes publish their TCP address to Nostr relays as application-specific events (kind 30078, d-tag pfm3-node). Other nodes subscribe and connect directly over TCP. No custom discovery infrastructure. Same secp256k1 curve as Bitcoin.

Wire Protocol

Binary FramingProtocol
[4 bytes: "PFM3"]  [4 bytes: payload length]  [JSON payload]
MessagePurpose
HANDSHAKE / HANDSHAKE_ACKIdentity exchange + challenge-response auth
PING / PONGKeepalive (30s interval, 90s timeout)
INV"I have these checksums"
WANT"Send me this document"
DATADocument bytes (base64 in JSON)
ANCHOR_ANN"This checksum was anchored in this txid"
NOT_FOUND"I don't have that document"
PEERS_REQ / PEERS_RESPeer exchange

Sync Cycle

Document ExchangeProtocol
Node A                          Node B
  |                                |
  |-------- INV [checksums] ------>|
  |                                |  (checks local store)
  |<------- WANT [checksum] ------|
  |                                |
  |-------- DATA [document] ------>|
  |                                |  (validates checksum)
  |                                |  (stores document)

Authentication

Every connection begins with a challenge-response handshake. Both sides exchange random 32-byte challenges, each signs the peer's challenge with their secp256k1 private key, and the connection is established only if both signatures verify. No anonymous connections.

Progressive Disclosure

New peers receive limited inventory to prevent store exfiltration by Sybil attackers:

Trust ScoreChecksums Disclosed
New (score ≤ 1.0)10
Known (1.0 < score ≤ 5.0)100
Trusted (score > 5.0)Full inventory

Key Defenses

Anti-Amplification

  • Per-peer WANT rate limit: 10/sec
  • Per-peer outbound bandwidth cap: 2 MB/sec
  • Per-message-type rate limits

Anti-Sybil

  • Per-IP inbound connection limit: 2/IP
  • Per-peer pending wants cap: 1,000
  • IPv4-mapped IPv6 normalization
  • Progressive inventory disclosure

Anti-Injection

  • DATA source validation
  • NOT_FOUND source validation
  • Anchor overwrite protection
  • Unsolicited DATA rejection

Architectural Firewall

  • No P2P message handler can trigger a Bitcoin RPC call
  • P2P layer and RPC layer not wired at runtime
  • Anchor verification injectable only at construction time

Certification

The system is certified for trusted-network beta deployment. Remaining items for public internet deployment:

  • TLS encryption on TCP connections
  • Handshake channel binding (MITM prevention)
  • Node key encryption at rest

VeilCloud: Privacy Primitives for L3

VeilCloud provides native Python privacy primitives built directly into Bitcoin L3. Client-side encryption, threshold crypto, Merkle audit trails, and privacy-preserving credentials — all built-in, zero external services.

Client-Side Encryption

  • AES-256-GCM — server never sees plaintext
  • PBKDF2-HMAC-SHA256 key derivation (600K iterations)
  • Best-effort key zeroing after use
  • l3 veilcloud encrypt/decrypt

Threshold Key Management

  • Shamir’s Secret Sharing over GF(256)
  • Same field as AES (Rijndael polynomial)
  • t-of-n: any t shares recover the key, fewer reveal nothing
  • l3 veilcloud split-key/combine-key

Merkle Batch Anchoring

  • Anchor thousands of documents in a single transaction
  • Domain-separated hashing (prevents second-preimage)
  • Inclusion proofs for any document in the batch
  • POST /batch-anchor + GET /merkle-proof

Audit & Access Control

  • Append-only hash-chained audit logs
  • Merkle proofs over audit entries
  • HMAC-signed credentials with expiry + revocation
  • Permission bitfield: READ, WRITE, DELETE, SHARE, ADMIN
Batch Anchoring FlowProtocol
1. COLLECT    N checksums from L3 store
2. BUILD      MerkleTree.from_checksums(checksums)
3. ANCHOR     Single OP_RETURN with Merkle root (36 bytes)
4. PROVE      tree.get_proof(index) → inclusion proof for any document
5. VERIFY     verify_proof(proof) + verify anchor on L1

Result:
  1,000 documents anchored in 1 transaction
  Cost: ~440 sats total (not per document)
  Each document independently provable via Merkle inclusion proof

Store & Anchor a Document

Basic Usagebash
# Store a document in L3
l3 store document.pfm
# Stored document.pfm in L3
#   checksum: a1b2c3d4e5f6...

# Anchor to Bitcoin (requires running Bitcoin node)
l3 anchor document.pfm \
  --rpc-url http://127.0.0.1:18332 \
  --rpc-cookie ~/.bitcoin/testnet3/.cookie

# Verify on-chain
l3 verify-anchor document.pfm --rpc-url http://127.0.0.1:18332
# OK: document.pfm anchor verified on testnet

The Strategic Vision

L3 isn't just software. It's an answer to the inscription civil war — a way to give data users what they want without forcing every full node to store their files forever.

The Core Question

Does L3 need L1 changes to win?

Option A: Don't Fight L1

Accept inscriptions exist. Build L3 as alternative. No political battle. No enemies. Ship faster.

Wins if: L3 adoption high enough that inscriptions become niche.
Fails if: People prefer L1 for "true immutability" status.

Option B: Win on Economics

Make L3 10x cheaper and better. Inscriptions die from irrelevance, not restriction.

Wins if: Cost difference is significant and UX is equal or better.
Fails if: "On-chain" has status that "anchored" doesn't.

Option C: Full Plan (L3 + L1 Changes)

Build L3, then push for witness restrictions via BIP. Belt and suspenders.

Wins if: L3 works AND L1 restrictions pass.
Fails if: L3 works but L1 never changes. Political effort wasted.

The Key Cultural Question

"I inscribed my art on Bitcoin" = status.
"I anchored my art to Bitcoin" = ???

If the market sees these as equivalent → L3 wins on cost.
If "inscribed" has magic status → L3 is second-class forever.

This might be the whole game.

The Reframe

L3 offers things L1 can't:

  • Larger files — L1 has limits, L3 doesn't
  • Updateable metadata — L1 is frozen, L3 allows revisions with new anchors
  • Collections/galleries — L3 could have native organization
  • Search/discovery — L3 could index, L1 can't
  • Cheaper minting — Artists could afford more
Don't build L3 as "the thing we need before we can fix L1."
Build L3 as "the thing that makes L1 inscriptions look stupid."

If L3 wins on merit, L1 changes become obvious.
If L3 doesn't win on merit, L1 changes wouldn't have saved it anyway.

The Four-Way Debate

Four adversarial positions stress-testing the strategy:

Option A: "Build Arks, Not Die on Hills"

"Satoshi didn't petition banks to change. He built an alternative. Fighting L1 is fighting physics—miners follow incentives, always will."

"We don't preserve ideals by dying on hills. We preserve them by building arks."

Option B: "Pick the Weapon That Works"

"We didn't end child labor through moral awakening—we made it economically obsolete. By making inscriptions economically irrational, we defend the vision with the same tool that created it: superior incentive design."

"Moral outrage without viable strategy is just screaming at the tide."

Option C: "The Only Answer That Doesn't Turn Its Back"

"Building an L3 while surrendering L1 is building a lifeboat while the ship burns—and pretending that's victory. L3 provides the carrot. Witness restrictions provide the stick. Neither alone is complete."

"Build the L3. Fight for L1. Accept nothing less."

The Skeptic: "Noble Cause. Poor Odds."

"History is brutally clear: moral arguments lose to money. 'Should' has never beaten 'profitable' at scale. The system has momentum measured in hundreds of billions of dollars. Your life has finite years. The asymmetry is severe."

"Are you fighting because you can win, or because you need to have fought?"

Content Accountability: The Channel Model

L1 inscriptions have a fatal flaw: illegal content stored on-chain lives in every full node forever, with no way to remove it. L3 solves this the same way Lightning solves payments — nothing hits L1 except a settlement hash.

L1 vs L3 Content ModelComparison
L1 Inscriptions:
  Content → embedded in witness data → every full node stores it forever
  Removal? Impossible. It's in the chain.

L3 Anchoring:
  Content → L3 node store (off-chain) → nodes choose what to store
  L1 gets: 36-byte hash only (reveals nothing about content)
  Removal? L3 nodes drop it. Hash on L1 becomes a dead pointer.

Built-in Accountability via Bitcoin Address

Every anchor transaction has a funding address. That address is public, timestamped, and permanent on L1. This creates an audit trail that L1 inscriptions lack:

Accountability ChainTrace
Illegal content detected on L3
    → checksum known
    → anchor txid found in L3 index
    → anchor tx has a funding Bitcoin address
    → address traced to exchange / KYC endpoint
    → identity of the person who anchored

Result:
  Content removed from L3 nodes (operator choice)
  Anchor hash remains on L1 (meaningless without data)
  Funding address is permanently linked to the act
  Better accountability than L1 inscriptions, where content is anonymous AND irremovable

What L3 Nodes Control

  • Store/don't store — every node chooses what content to hold
  • Relay/don't relay — nodes can refuse to propagate specific checksums
  • Blocklists — community-maintained lists, like email spam filters
  • No content ever touches L1 — the hash is just a number

What L1 Provides

  • Anchor address = permanent record of who anchored what
  • Timestamp = when it happened
  • Hash = cryptographic link (but reveals zero content)
  • The chain of custody is better than inscriptions
L1 inscriptions: content is irremovable, uploader is anonymous.
L3 anchoring: content is removable, uploader is traceable.

The Lightning model applied to data: off-chain until settlement, and even at settlement only a hash hits the chain.

Node Content Policies: Opt-In by Category

Not every node stores everything. L3 node operators choose what categories of content they accept, the same way you choose what apps to install on your phone.

~/.pfm/l3/node.tomlConfig
# Node content policy — operator chooses what to store and relay

[content_policy]
accept = ["legal", "record"]     # only store these categories
reject = ["*"]                     # reject everything else
max_document_size = "50MB"         # per-document cap
require_category = true            # reject untagged documents

Different nodes, different purposes:

Legal Archive Node

  • Accepts: legal, record
  • Wills, contracts, deeds, corporate filings
  • Run by: law firms, notary services, estate planners
  • Business model: paid archival service

Art & Media Node

  • Accepts: art, media, nft
  • Photography, music, digital art, collections
  • Run by: artists, galleries, NFT platforms
  • Business model: hosting + discovery

Personal Storage Node

  • Accepts: personal, media
  • Family photos, personal records, memories
  • Run by: individuals, families
  • Business model: none — you run it because it's yours

General Purpose Node

  • Accepts: everything (or broad categories)
  • Public archive, research data, open records
  • Run by: archivists, researchers, enthusiasts
  • Business model: community service, grants, donations

Report & Delete Workflow

Content FlaggingWorkflow
1. FLAG     Any node operator flags a checksum
2. REVIEW   Each node operator reviews flagged content independently
3. DELETE   Operator removes content from their store: l3 delete <checksum>
4. RELAY    Flag propagates to peers — each peer makes their own decision
5. TRACE    Anchor tx on L1 links checksum → funding address → identity

Result:
  Content gone from L3 (every node that agreed to delete)
  Hash on L1 = dead pointer (proves nothing without the data)
  Uploader traceable via anchor address
  No central authority decided. Every node chose independently.

Why People Run Nodes

The same reason people run Bitcoin full nodes: because they want to use the network.

Nobody pays you to run a Bitcoin node. You run it because you need it. L3 is the same. Want to anchor legal documents? Run a node. Want to host your art? Run a node. Want a family photo archive with Bitcoin-grade proof of existence? Run a node.

Two nodes find each other via Nostr. They mesh. Now there's redundancy. That's it. The incentive is using the thing. The network grows because people need what it does.

Bitcoin is the most secure timestamping system ever created. Using it to store data is like using a bank vault to hold Post-it notes — it works, but it wastes the vault's capacity and makes everyone's safe deposit box more expensive.

36 bytes per document. Permanent proof. Zero bloat.
